# ジャーナル記事の執筆ルール

**書く前に読み、書いた後にチェックリストとして通す**。手続き（下書きページ・画像・公開）は `journal-publish` スキル側。

## 基本設計

- 文体: です・ます調。少し詩的、余韻を残す。前撮り体験の物語性を大事にする
- 文字数目安（公開済み全記事の実測・2026-07-07）: **ロケーション記事400〜600字／知識・ガイド系500〜1000字**（「準備すること」のような網羅リスト型のみ〜1500字可）。CTA・「← 戻る」は含まない
- 末尾CTAは記事タイプで使い分け: ロケーション記事＝3行CTA（柱①のテンプレ参照）／知識・ガイド系＝1行CTA「名古屋・愛知を中心に前撮り・フォトウェディングのご相談はLINEからどうぞ。→ https://lin.ee/OYalLp6」
- 改行: 1文＝1段落。**段落と段落の間は必ず空行で区切る（単一改行で文を連ねない）**。理由: サイトは `marked` が空行を段落境界として解釈するため、空行がないと複数文が1段落に結合する。またObsidianのリーディングビューでも単一改行はソフト改行として詰まって表示される。下書き（vault）も公開記事と同じ空行区切りで書くこと（2026-07-09、救出下書きが単一改行で詰まっていたのを全整形した）
- モバイルは `、` の直後に自動改行（`comma-br`、実装済み・追加作業不要）
- 章タイトルは `##`（heading_2）。全記事共通デザインに統一する（デザイン分岐は作らない）
- 固定キーワード「**愛知 / 名古屋 / 前撮り / フォトウェディング**」を自然に含める
- 本文にお客様の実名は載せない（タイトルの「Y & H」のようなイニシャル表記のみ）
- ロケーション記事のタイトルはロケーション名のみ（余計な説明・詩的表現なし）。複数ロケーションは「/」区切り（例:「葦毛湿原 / 伊古部海岸」。✕「愛知・葦毛湿原と伊古部海岸、ふたりの前撮りを紡ぐ」）。知識・ガイド系記事はテーマ名＋検索されそうな主要語を含める（「小物アイデア」より「前撮りの小物・アイテムのアイデア」）。絵文字プレフィックス（📋）は過去記事にあるが**新規では付けない**
- 言い切らない。文末・見出しは「〜と、」「〜ていきました。」のような開いた表現で締める（✕「これ以上ないと思いました。」→ ○「ベールがなびいて、光がオレンジに染まる。」）
- **本文に画像を必ず散らす**（見出しセクションの区切りに1枚ずつが目安。画像URLは `https://www.inthemoment.jp/gallery/case-XX/...jpg` の本番URL）。画像0枚の記事は公開不可（「雨の日の前撮り」が画像なしで公開保留になった実例。公開済みの知識系3本に画像0のものがあるがそれは旧基準 — 真似しない）
- **全ケース（case-01〜06）の画像を使ってよい**。ファイル名にお客様の下の名前が入っているもの（case-02/05/06の一部）も使用可 — フルネームではなく掲載同意済みのため許容とユーザー判断（2026-07-07）。ただし**スペース入りファイル名はURLエンコード（`%20`）が必要**な点だけ注意。今後の新ケースは `gallery-case-add` スキルが配置時に `case-XX-NN.jpg` へリネームするのでこの問題自体がなくなる
- **画像はReadツールで実際に開いて中身を見て選ぶ**（サブエージェントでも可能と実証済み）。セクションの内容に合う写真を選び、それでも最終確認はユーザーが行う（差し替え前提でレビュー依頼する）
- 候補が多い時は**フルサイズを直接開かず、320pxサムネイルを作って見る**（`sips -Z 320` でscratchpadへ生成。コスト約1/5で同じ判断ができる。case-07の110枚セレクトで確立・2026-07-08）
- **カバー・本文画像とも、他の記事（下書き含む）で使用済みの画像は使わない**。特にカバーの重複は一覧ページで並んだ時に目立つ（テストでcase-01/DSCF2631.jpgがカバー重複した実例）
- **使用済み画像の洗い出し手順（画像選定の前に必ず実行）**: ✅公開中と📝下書きの全記事を `retrieve-page-markdown` で取得し、`![](...)` のURLとカバーURLを一覧化 → その一覧に**ない**画像から選ぶ。プランページ使用分（`app/plan/data.ts` の tierImage/midImage）も避けると理想的
- **書き出しの重複禁止**: 既存記事と同じ書き出しにしない（「名古屋・愛知で前撮りを考えているふたりへ。」が2記事で重複していた実例）。書く前に既存記事の冒頭を確認する
- **比喩・締め文などの言い回しも記事間で重複禁止**: 書き終えたら `node .claude/skills/journal-publish/dump-articles.mjs > /tmp/articles.txt` で全記事本文を出し、自分が書いた特徴的なフレーズ（比喩・情景の締め・「〜が〜に溶ける/混ざる」のような型）をgrepして照合する。実例（2026-07-08検出）: 「笑い声が〜に溶けて/混ざって」の型が4記事で反復・「なんでもない〜が、ふたりだけの景色に」が豊田市記事とほぼ同文で再生産された。**同じ情景でも、写真に写っている別の要素から言葉を起こす**と自然に散る
- 関連する公開済み記事があれば本文から内部リンクを張る（`https://www.inthemoment.jp/journal/<slug>`。SEOの内部リンク効果）
- カバー＝Notionページのcoverプロパティに本番画像URLを外部URLとして設定（これがヒーロー画像になる）。カバーに使った画像は本文から削除する（重複禁止）
- ロケーション記事の本文1行目（「← 戻る」の直後）に都道府県・市区町村を入れる（例:「愛知県豊橋市にある葦毛湿原と伊古部海岸。」）

## 3つのコンテンツ柱

サイトのSEO戦略（LIGHTLEAKと共通方針）に基づく3本柱。①は撮影ごと、②③は在庫写真だけで書けるので撮影がなくても量産できる。

### ① ロケーションレポート（実撮影ケースのストーリー）

実際の撮影ケース1件＝記事1本。`app/gallery/data.ts` のcaseと1:1対応。

**現状（2026-07-07時点）**: case-01〜06 すべて記事化済み（6本）。今すぐ書ける新規ネタはない — **新しいケースを追加した時のみ**新規記事が発生する（`gallery-case-add` スキルでケース追加 → 本スキルで記事化）。

**構成テンプレ（公開済み6本の実測から抽出・2026-07-07）**:
1. タイトル＝ロケーション名（複数は「/」区切り）
2. 本文1行目＝都道府県・市区町村（「愛知県知多郡南知多町の篠島。」）
3. 導入1〜3文（その日の始まり・空気感）
4. 見出し2〜3個＝**時間・場所の移動単位**で切る。新しめの記事は「場所名 — 情景フレーズ」形式（「豊田市美術館 — 光と影のなかで」「東岡崎 — 夕日の街を歩いて」）
5. 各セクション＝情景描写3〜5文 → **画像2〜3枚連続**（画像は文中に散らすのではなくセクション末にまとめる）
6. 文字数400〜600字・画像計4〜9枚。描写は「〜ていきました」で流し、撮影者の一人称視点で書く
7. **締めはロケーション記事専用の3行CTA**（知識系の1行CTAとは別物）:
   > in the momentでは、名古屋をはじめ愛知県内・県外への出張撮影も承っています。
   > 前撮り・フォトウェディングのご相談は、お気軽にLINEからどうぞ。
   > LINEでは、季節の撮影のお知らせや、お得なご案内も、ときどきお届けしています。

### ② 前撮り知識・Q&A（検討フェーズの疑問に答える）

**既存（公開済み）**: 「前撮りとフォトウェディングの違い」「前撮りの費用と相場」
**下書き済み（2026-07-07）**: 「ロケーション前撮りとスタジオ撮影」「式を挙げない、という選択」

**書ける候補**（着手前に下の「事実確認」と既存記事との重複チェックを必ず通す）:
- 前撮り、いつから準備を始める？（逆算スケジュール。※「前撮り、準備すること」の衣装2〜3ヶ月前の話と重複しない切り口にする）
- 衣装チェンジは何着まで？（プランごとの相場感。※プラン内容の事実確認必須）

※「前撮りのベストシーズン」は候補から削除済み — 公開済みの「📋 季節の、選び方。」が同テーマだった（候補を挙げる時は必ず公開済み記事の本文まで確認する）

### ③ 撮影準備ガイド（実用的・シェアされやすい）

**既存（公開済み）**: 「前撮り、準備すること」「📋 季節の、選び方。」「📋 緊張している、あなたへ。」
**下書き済み（2026-07-07）**: 「前撮り当日の流れ」（plan/data.tsの実タイムテーブルを使用）
**下書き中（画像0枚・要修正のため未公開）**: 「雨の日の前撮り」— 公開前に本文中に画像を1〜2枚追加する

**書ける候補**:
- 自然な表情・ポーズの作り方（※「📋 緊張している、あなたへ。」と重複しないか本文確認してから）
- 小物・アイテムのアイデア（花束・お手紙・ふたりの思い出の品）
- 撮影後の写真の活用方法（アルバム・招待状・SNS投稿）

## 事実確認が必須の項目（書く前に確認）

- 価格・プラン内容は `app/plan/data.ts` と実際のプランページを必ず参照する（page.tsxとの二重定義バグの実績あり）
- 提供していないサービス・存在しないプランを書かない
- **実例のない体験談を書かない**: 「〜する方がいます」「これまでの撮影で〜」のような実例を示唆する表現は、実際にあったとユーザーに確認できた場合のみ。確認していないアイデアは「〜もおすすめです」の提案形で書く（小物記事の手紙セクションで「手紙を読む方がいます」と書いて指摘された実例・2026-07-07）
- 迷ったらユーザーに確認してから書く（相場観・提供有無など事実に関わる部分はAIの推測で書かない）

## 公開前チェックリスト（書き終えたら機械的に通す）

- [ ] 固定キーワード4つ（愛知・名古屋・前撮り・フォトウェディング）が本文に自然に入っている
- [ ] 書き出しが既存記事と重複していない（直近数本の冒頭と照合した）
- [ ] 比喩・締め文が既存記事と重複していない（`dump-articles.mjs` の出力に対して特徴フレーズをgrepした）
- [ ] 実例を示唆する表現（「〜する方がいます」等）は確認済みの事実のみ
- [ ] 価格・プラン内容を書いた場合、`app/plan/data.ts` と一致している
- [ ] 本文に画像が入っている（見出しセクション区切りに1枚目安・0枚は不可）
- [ ] 画像・カバーとも他記事（公開＋下書き）と重複していない（`list-used-images.mjs` で照合した）
- [ ] カバーに使った画像が本文にない
- [ ] お客様の実名が本文にない
- [ ] 章タイトルがheading_2になっている・1文＝1段落・セクション間に空行
- [ ] 締めが言い切り・指示語まとめになっていない
- [ ] 先頭に「← 戻る」・ロケーション記事なら本文1行目に都道府県・市区町村
- [ ] 末尾にLINE CTA・関連記事があれば内部リンク

## Notion保存後の必須チェック

日本語はリテラルUTF-8で書く（`\u`エンコード禁止。文字化け実績あり）。保存後は `mcp__notion__API-get-block-children` で読み返して文字化け・誤字がないか確認する。
